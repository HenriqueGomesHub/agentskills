# Claude Skills

My personal collection of [Claude skills](https://docs.claude.com/en/docs/build-with-claude/skills). Each skill lives as an editable folder under `skills/`, and gets built into a single-file `.skill` archive under `dist/` for installation.

## Repo layout

```
claude-skills/
├── skills/                       # editable source folders (the truth)
│   ├── anti-slop/
│   │   ├── SKILL.md
│   │   ├── references/
│   │   ├── scripts/
│   │   └── hooks/
│   └── senior-dev/
│       ├── SKILL.md
│       ├── references/
│       └── assets/
├── dist/                         # built .skill archives (drag-and-drop install)
│   ├── anti-slop.skill
│   └── senior-dev.skill
├── scripts/
│   └── build_all.py              # builds dist/ from skills/
└── README.md
```

`skills/` is what you edit. Git tracks meaningful diffs there. `dist/` is the install-ready output, regenerated by the build script whenever a source folder changes.

## Skills in this repo

### `anti-slop`
Forces clean output. Suppresses filler phrases, unsolicited summaries, hallucinated APIs, and other LLM noise. Active on every prompt with substantive output.

### `senior-dev`
Applies senior-engineer discipline (Clean Code, Refactoring, Pragmatic Programmer, Clean Coder, Code Complete) to all coding tasks. Pragmatic mode — ships what you ask for, with brief heads-ups on tradeoffs. Multi-stack: React, Express, TypeScript/JS, Python, plus general guidance for other languages.

## Building skills

Requires Python 3.9+. No external dependencies.

```bash
# Build every skill
python scripts/build_all.py

# Build one skill by name
python scripts/build_all.py senior-dev
```

Each build:
1. Validates the skill's `SKILL.md` frontmatter (name present, description ≤ 1024 chars, name matches folder)
2. Zips the folder into `dist/<name>.skill`

If validation fails, the skill is skipped and the error is printed.

## Installing a skill in Claude

Either:
- **Drag the `.skill` file** into Claude's Skills settings, **or**
- **Drag the source folder** in (Claude unpacks it the same way)

Both produce identical results internally.

## Adding a new skill

1. Create a new folder under `skills/<your-skill-name>/`
2. Add a `SKILL.md` with frontmatter:
   ```markdown
   ---
   name: your-skill-name
   description: One paragraph (max 1024 chars) describing when to trigger this skill. Be specific about the verbs, contexts, and prompt patterns that should activate it. Aggressive triggering language ("trigger on...", "do not skip for...") is encouraged for skills that should run frequently.
   ---

   # Your skill body here
   ```
3. Add reference files under `references/`, assets under `assets/`, scripts under `scripts/`, hooks under `hooks/` — only what you need.
4. Run `python scripts/build_all.py your-skill-name` to validate and build.
5. Commit and push.

## Skill design notes

A few things I've learned that apply to every skill in this repo:

- **The `description` field is the trigger.** Claude decides whether to load the skill body based on the description alone. Be aggressive and explicit about triggers — list verbs, contexts, prompt patterns. Don't be subtle; subtlety produces under-triggering.
- **Keep `SKILL.md` lean.** It's loaded into the context window every time the skill activates. Push detail into `references/` and load on demand.
- **`references/` is for progressive disclosure.** Add a routing table to `SKILL.md` that tells Claude which reference to load based on the task. Don't dump all references into context.
- **`assets/` is for pasteable artifacts.** Checklists, templates, decision trees that the skill might surface in a response.
- **`scripts/` and `hooks/` are for active enforcement.** Rare; use only when behavior can't be expressed declaratively.
